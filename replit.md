# Masah Boutique — بوتيك ماسـة

## Overview
Masah Boutique is a full-stack e-commerce application featuring a boutique with a Flutter mobile frontend and an Express.js backend. The project aims to provide a luxurious online shopping experience with a focus on high-quality fashion items. It supports a wide range of features including product browsing, cart management, order placement, and discount validation.

The Khurais Emergency Alert System (KEAS) is an emergency alert and personnel tracking system designed for the Khurais industrial facility. It provides a robust platform for managing emergency situations, tracking personnel, and ensuring timely communication. The system is currently a demo/prototype, with all data managed in a Zustand store with localStorage persistence.

## System Architecture

### Masah Boutique
- **Backend**: Express.js with TypeScript, Drizzle ORM for PostgreSQL. Handles API routes for products, cart, orders, stores, and discounts.
- **Frontend**: Flutter mobile application.
- **Database**: PostgreSQL via Drizzle ORM, schema defined in `artifacts/boutique-server/shared/schema.ts`. Auto-seeds initial data.
- **UI/UX (Theme)**: Dark luxury theme (`#1A1A2E` background, `#C8A96E` gold, `#222240` card).
- **Localization**: Bilingual support for Arabic (RTL) and English.

### Khurais Emergency Alert System (KEAS)
- **Frontend**: React with Vite, utilizing Zustand for state management.
- **UI/UX**: Dark theme, responsive layout (desktop table to mobile card at `lg` breakpoint). Mobile app uses a light enterprise palette (`#F5F6F8` background, white cards, purple header/accents) with platform-aware tab bars.
- **State Management**: Zustand v5 with persistence to `localStorage`. Selectors that return objects/arrays MUST cache results (module-level memoization with input key) to satisfy React 18's `useSyncExternalStore` requirement for stable `getSnapshot` returns. See `selectAlertSystemState` pattern in `store/selectors.ts`. The `useAlertSystemState` hook is the single source of truth for all alert/emergency state across screens.
- **Mapping**: Web preview uses Leaflet iframe with Google Maps tiles via GoogleMutant plugin when `EXPO_PUBLIC_GOOGLE_MAPS_API_KEY` is set; falls back to Esri World Imagery if no key. Supports 4 map types (satellite, hybrid, standard, terrain/dark) via floating layer switcher. Google Maps-style floating controls (zoom, locate, layers) on right side. Shelter clustering at zoom < 14, zoom-dependent labels (show at zoom >= 15 for shelters, >= 13 for zones). Zone polygons use subtle fills (0.1 opacity) with selected-zone highlighting. Native targets Google Maps with `react-native-maps`. Both `EXPO_PUBLIC_GOOGLE_MAPS_API_KEY` and `VITE_GOOGLE_MAPS_API_KEY` secrets must be set.
- **Map Overlay Layout**: `MapOverlayLayout` component (`components/map/MapOverlayLayout.tsx`) defines fixed layout zones for all floating map UI elements. Top-left zone: alert banner, INTEL bar, timeline. Top-right zone: wind indicator, wind control. Bottom-center: legend/stats. Bottom-action: all-clear button. Shared `MAP_UI` spacing constants (topPad, sidePad, gutter, bottomPad). Components (`SmartAlertPanel`, `WindIndicator`, `MapLegendCounts`, `IncidentTimelinePanel`) no longer use absolute positioning — all positioned by the layout wrapper. Z-index hierarchy managed by layout. No element overlap. Used by ECO live-map and Admin alert-monitor.
- **Location Awareness**: `utils/geo.ts` for `pointInPolygon`, `haversineDistance`, `findBestShelter`. `useDetectedLocation.ts` for GPS-based location detection.
- **Personnel Tracking**: Live personnel tracking activated during active alerts. Polls GPS (configurable interval), simulates other users for demo. Displays personnel on maps with color-coded status.
- **Authentication & Authorization**: Badge and password login. Role-based access (Super Admin, IT, ECO, Supervisor, User). Registration with approval workflow, including role override and rejection reasons. Route guards enforce authentication and authorization across all layouts.
- **Store**: `keas-mobile-store-v20` (persist key), version 32 with slice architecture. Includes domain-specific slices, user management (approval status, user types), emergency-related state, accountability session slice, streets slice, and routes slice.
- **Street Segments**: Internal CPF routing streets rendered as polylines on the map. `Street` type with `id`, `name`, `path` (LatLng[]), `createdAt`, `updatedAt`. Store slice at `store/slices/streets.ts` with addStreet, updateStreet, deleteStreet, duplicateStreet. Admin screen at `app/(admin)/streets.tsx` with draw mode, path editing, rename, delete, duplicate. Map rendering via postMessage protocol (sync_streets, select_street, start_street_edit, clear_street_edit, set_street_draw_mode). Nav link in admin settings under Navigation section.
- **ECO Route Builder**: ECO can build emergency routes by tapping existing streets in sequence during active alerts. Route builder accessed via "Route Mode" FAB on ECO live-map. Selected streets turn green (#10B981) with thicker stroke. Connection validation ensures each new street connects to the previous one (endpoint proximity threshold 0.0003°). Controls: Start Route, Undo Last, Clear, Confirm. Shows route length in meters/km. `EcoRoute` type in `store/slices/routes.ts` with `streetIds`, `createdBy`, `status` (active/edited). Routes persisted to store (STORE_VERSION=32). Map rendering via `routeStreetIds` prop and `set_route_streets` postMessage.
- **Accountability System**: DB-backed accountability sessions with API persistence. `accountability_sessions` + `personnel_status` tables in PostgreSQL. REST API at `api-server/api/accountability` (start/respond/end/session/active/escalate/history). Zustand accountability slice syncs with API, provides live timer and polling. Supervisor dashboard shows live session panel with counters, progress bar, and personnel status list. API uses transactions for session start, validates session lifecycle (active-only respond/end), and prevents duplicate active sessions per location. Session continuity: supervisor dashboard auto-resumes active sessions on mount via `resumeAccountabilitySession`.
- **Self-Status Bar**: `SelfStatusBar` component (`components/ui/SelfStatusBar.tsx`) allows Supervisor and ECO roles to set their own safety status during active alerts. Shows "I am Safe" / "Need Help" buttons when no response given, compact status with change option after responding. Uses `respondToAlert` from alerts slice. `mobileUserResponse` is reset to `null` on all emergency entry paths (createAlert, activateZoneAlert, activateShelterIn, activateBlackout, bulkActivateZoneAlerts).
- **Warning Zones (Hazard Zones)**: Standalone feature decoupled from alerts. Admin places warning zones from floating tool button on zone map (tap Warning → tap map → confirm). All 3 rings (hot/warm/green) created automatically using Settings radii. Circles rendered in Leaflet (hot=red, warm=yellow, cold/green). `HazardZone.alertId` is nullable — standalone warnings persist across alert lifecycle. `WarningLevel` type: `'hot' | 'warm' | 'green'`.
- **Zones**: CPF (id=1, industrial) and Camp (id=2, residential). Camp zone has 16 personnel with mixed Aramco/Contract types and confirmed/pending/need_help statuses. All locations have polygon boundaries. Zones support `sortOrder` (display ordering), `isArchived` (soft archive — hides from active lists, preserves all data). Archived zones excluded from alert creation, ECO assignment, emergency zone selection, and zone breakdowns.
- **Zone Management**: Archive = main action (non-destructive), Delete = restricted via `safeDeleteZone` (blocked if zone has locations, shelters, users, alerts, hazard zones, or alert history). Locations also have `sortOrder` for manual ordering within their zone. Reordering is display-only — no impact on alert logic or tracking.
- **Migration safety**: Migrations are ADDITIVE — they merge missing seed entities by ID, never overwrite existing user-created zones/locations/users/shelters. Pattern: check `existingIds` Set, only push if missing.
- **Filter Chips**: Alert Monitor legend chips filter by status (Safe/Pending/Help) and personnel type (Contract). Uses `userType === "Contract"` (matching `UserType` enum), not `"Contractor"` (which is `CompanyType`).
- **Alert System**: Comprehensive alert lifecycle (activate, clear, reactivate). Includes zone-specific alerts, shelter-in-place, and blackout modes.
- **Emergency Sound & Vibration**: `useEmergencyAlerts` hook plays Web Audio oscillator siren (web) or expo-av WAV (native) plus vibration pattern during active emergencies. Auto-stops on user response or alert clear. Mounted via `SystemServices` component in root layout.
- **Escalation System**: `useEscalation` hook auto-promotes unresponsive users: Level 0→1 (warning, after timeout), 1→2 (critical, after 2×timeout), need_help=Level 3 (immediate). Timer-based checks every 30s during active emergencies. `alertReceivedAt` set on all emergency types (broadcast, zone, shelter-in, blackout). Reset on all-clear.
- **Notification Triggers**: Admin/ECO/Supervisor roles receive local notifications on escalation level changes and immediate `need_help` submissions via store subscription. Settings page controls alert sound, push notifications, and escalation timeout (1–30 min stepper).
- **Map Escalation Visualization**: Leaflet personnel dots add `.escalated` (amber glow pulse) and `.critical` (red glow pulse) CSS classes based on `escalationLevel` from `useVisiblePersonnel`.
- **Users Escalation UI**: User cards show "Critical" (red) and "Escalated" (amber) badges. Users sorted by urgency: need_help → escalation level → status → name.
- **Incident Timeline + Replay**: `IncidentEvent` type in `types/index.ts` with 14 event types. `incidentTimeline` store slice (`store/slices/incidentTimeline.ts`) with `logIncidentEvent` / `clearIncidentTimeline` actions and 500-event cap. Events auto-captured from: `createAlert` (alert_started), `sendAllClear` (all_clear), `respondToAlert` (user_safe/user_need_help), `activateZoneAlert`/`deactivateZoneAlert` (zone_updated), `activateShelterIn` (shelter_in_activated), `activateBlackout` (blackout_activated), escalation hook (escalation_level_1/escalation_critical). `useIncidentReplay` hook (`hooks/useIncidentReplay.ts`) provides play/pause/reset/scrubTo/setSpeed (1x/2x/4x/8x) with progress tracking. `IncidentTimelinePanel` (`components/ui/IncidentTimelinePanel.tsx`) — collapsible panel with Live/Replay mode toggle, full timeline view with event dots/lines, replay controls with progress bar and speed selection. Wired into Admin alert-monitor, ECO live-map, and Supervisor dashboard.
- **Intelligence Layer**: Decision-driven situation awareness system. `useEmergencyIntelligence` hook (`hooks/useEmergencyIntelligence.ts`) provides zone/location-level intelligence with missing counts, need_help counts, severity ranking, and auto-generated suggested actions. Scoped by type: `"all"` (admin), `"zone"` (ECO), `"location"` (supervisor). `SmartAlertPanel` (`components/ui/SmartAlertPanel.tsx`) is a floating, collapsible overlay for map-centric screens (ECO live-map, Admin alert-monitor) showing suggested actions and affected areas. Supervisor dashboard uses inline intelligence card. Map supports `focus_critical_users` / `clear_intel_focus` postMessage commands with persistent `intelFocusedIds` set that survives personnel sync. Panel action taps focus map on critical personnel via existing `trackedUserIds` + `fitTrackedTrigger` props.
